# kongctl command docs skill

Generates reference pages and index updates for new kongctl commands, using pre-generated include files from `app/_includes/kongctl/help` as the source of truth.

## Invocation

**Option A: Pass a GitHub PR link for automatic discovery**

Use this when there is an open "Sync kongctl Releases" PR opened by the GitHub Action. The skill will discover which commands are new by inspecting the PR:

```
/kongctl-new-command https://github.com/Kong/developer.konghq.com/pull/<PR-NUMBER>
```

Start at Step 1.

**Option B: Pass a command path directly**

Use this when you already know which commands need pages — either because the PR has already been merged, or you want to generate a specific subset. Pass a file or glob relative to `app/_includes/kongctl/help`:

```
/kongctl-new-command get/event-gateway/*
/kongctl-new-command get/event-gateway/backend-clusters.md
```

Skip to Step 2.

## Prerequisites

You need `kongctl` installed to fetch command descriptions. Check whether it's available:

```sh
which kongctl && kongctl version
```

If it's not installed, ask the user before proceeding:

```sh
brew install kongctl   # or: brew upgrade kongctl
```

The agent will only run kongctl commands with the `--help` flag.

## Step-by-step instructions

### Step 1: Identify new commands from the GitHub Action PR

> Only needed for Option A. Skip to Step 2 if you used Option B.

The include files under `app/_includes/kongctl/help` are auto-generated by a GitHub Action that opens a PR titled "Sync kongctl Releases". The PR touches many files — both updates to existing commands and brand-new ones. You only want to create pages for the *newly added* files, not updated ones, since updated files already have pages.

Use `--paginate` to ensure all pages of results are fetched (the API returns at most 30 files per page, and these PRs often touch 100+ files):

```sh
gh api repos/Kong/developer.konghq.com/pulls/<PR-NUMBER>/files \
  --paginate \
  --jq '.[] | select(.status == "added") | .filename' \
  | grep "app/_includes/kongctl/help"
```

The output is the list of new include files. Strip the `app/_includes/kongctl/help/` prefix to get the `<COMMAND-PATH>` for each.

### Step 2: Create reference pages

For each new include file, create a matching `.md` page under the parallel `app/kongctl/` directory. For example, `get/event-gateway/backend-clusters.md` becomes `app/kongctl/get/event-gateway/backend-clusters.md`. Create any missing directories as needed.

Before creating pages, look at a few existing pages in the same directory to understand the established pattern — particularly the `works_on` values and `breadcrumbs` used there.

Fill out each page using this structure:

```yaml
---
title: <command name derived from directory path, e.g. "kongctl get event-gateway backend-clusters">
description: "<Short description of what the command does>"
content_type: reference
layout: reference

beta: true # kongctl is in beta, so all new pages need this flag; remove this flag when it reaches stable release.
works_on:
  # kongctl is a Konnect-native CLI. Set both on-prem and konnect only for pages
  # under /gateway/, since those commands interact with on-prem Gateway resources.
  # All other pages should be konnect only.
  - konnect

tools:
  - kongctl

breadcrumbs:
  # List each ancestor directory as a breadcrumb, derived from the page's path.
  # For example, app/kongctl/get/event-gateway/backend-clusters.md:
  - /kongctl/
  - /kongctl/get/
  - /kongctl/get/event-gateway/

related_resources:
  # Link to the nearest parent index page (the top-level verb index).
  - text: kongctl <verb> commands
    url: /kongctl/<verb>/
---

<Short description of what the command does, matching the description field above>

## Command usage

{% include_cached /kongctl/help/<path/to/include/file.md> %}
```

To find the description, run `kongctl <command> --help` and use the opening paragraph — the text that appears before "Usage:". The include files in `app/_includes/kongctl/help` do not contain this paragraph; they start directly with the `Usage:` block.

Before using the text, apply these formatting rules:
- Wrap verb names (such as `get`, `list`, `adopt`) in backticks when referenced inline
- Wrap subcommand names (such as `dcr-provider`, `event-gateway`) in backticks
- Replace "Konnect" with `{{site.konnect_short_name}}`
- Replace "Event Gateway" with `{{site.event_gateway_short}}`
- Replace "Kong Gateway" with `{{site.base_gateway}}`

If the opening paragraph is a single sentence, use it for both the `description` frontmatter field and the body text above `## Command usage`. If it is longer than one sentence, use only the first sentence for the `description` field, and the full paragraph for the body text. For example:

```sh
kongctl get event-gateway backend-clusters --help
# → "Use the `backend-clusters` command to list or retrieve backend clusters for a specific {{site.event_gateway_short}}."
```

### Step 3: Create index pages for new directories

For any directory that didn't exist before, create an `index.md` inside it. For example, if `app/kongctl/get/event-gateway/` is new, create `app/kongctl/get/event-gateway/index.md`.

The index page serves as the landing page for the command group and lists all its subcommands. Use this structure:

```yaml
---
title: <command group name, e.g. "kongctl get event-gateway">
description: "<Short description of what this group of commands does>"
content_type: reference
layout: reference

beta: true
works_on:
  - konnect  # same rules as Step 2

tools:
  - kongctl

breadcrumbs:
  # Same as the subcommand pages in this directory, derived from the directory path.
  - /kongctl/
  - /kongctl/<verb>/
  - /kongctl/<verb>/<command>/

related_resources:
  - text: kongctl <verb> commands
    url: /kongctl/<verb>/
---

<Short description of what this group of commands does>

kongctl provides the following tools for <what this group does, e.g. "retrieving resources and resource details for {{site.event_gateway_short}}">:

{% table %}
columns:
  - title: Command
    key: command
  - title: Description
    key: description
rows:
  - command: |
      [kongctl <verb> <command> <subcommand>](/kongctl/<verb>/<command>/<subcommand>/)
    description: "Short formatted description of the subcommand."
  # ... one row per subcommand in this directory
{% endtable %}

## Command usage

{% include_cached /kongctl/help/<path/to/index/include.md> %}
```

Run `kongctl <command group> --help` to get the opening description and the list of subcommands with their short descriptions. For example:

```sh
kongctl get event-gateway --help
# → "Use the `get` verb with the `event-gateway` command to query {{site.konnect_short_name}} {{site.event_gateway_short}}s."
# → Available Commands: backend-clusters, listeners, ...
```

Apply the same rules as Step 2: use only the first sentence for the `description` field if the opening paragraph is multi-sentence, use the full paragraph for the body text, and apply the same formatting (backticks for verbs and subcommands, template variables for product names). Use the subcommand list to populate the `{% table %}` rows.

### Step 4: Update parent index pages

Every new command needs to appear in its parent's command table, so users can navigate to it. For each new page (and each new index page), find the parent index and add a row.

For example:
- A new `app/kongctl/get/dcr-provider.md` → add a row to `app/kongctl/get/index.md`
- A new `app/kongctl/get/event-gateway/` directory → add a row to `app/kongctl/get/index.md`
- A new `app/kongctl/get/event-gateway/backend-clusters.md` → add a row to `app/kongctl/get/event-gateway/index.md`

Read the parent index before editing it, and insert new rows in alphabetical order by command name. Keep the description short.

### Step 5: Update the navigation index

`app/_indices/kongctl.yaml` is the site-wide navigation index for the kongctl section. It needs to be updated when a **new top-level command** is added (e.g., `explain`, `scaffold`). Subcommand pages under an existing verb are covered by the existing glob patterns (e.g., `/kongctl/get/*/**`) and do not require changes here.

For a new top-level command, read the file first, then add a section under "CLI References" in alphabetical order by verb name:

```yaml
      - title: <Descriptive section title>
        items:
            - path: /kongctl/<verb>/
            - path: /kongctl/<verb>/*/**    # add if verb has subcommand groups
            - path: /kongctl/<verb>/*/**/** # add if verb has 3+ level nesting
```

For example, `explain` (no subcommands) becomes:

```yaml
      - title: Explain Schema
        items:
            - path: /kongctl/explain/
```

### Step 6: Summarize and report to the user

List the following:
* New pages created
* Pages updated
* Commands that were checked and help text that was pulled from them
